Tengo un proyecto Python llamado SlideGuard. Monitorea una Google Slides presentation, detecta cambios slide por slide, y manda un alert a Slack/Google Chat con una interpretación semántica de qué cambió y por qué importa.
Adjunto todos los archivos del proyecto. Necesito que resuelvas exactamente estos 3 bugs en orden de prioridad, sin agregar features fuera de scope.

BUG 1 — Diff por índice en lugar de objectId (CRÍTICO, resolver primero)
Archivo: slide_diff.py
Problema: La implementación actual compara slides por posición/índice. Si alguien reordena slides, el sistema reporta falsos cambios masivos.
Cómo debe funcionar:
Reescribir completamente la lógica de diff para comparar exclusivamente por objectId. La lógica exacta es esta:

Slide eliminada: slide_object_id existe en el snapshot pero NO en la respuesta actual de la API.
Slide agregada: slide_object_id existe en la respuesta actual de la API pero NO en el snapshot.
Texto modificado: slide_object_id coincide AND text_element.object_id coincide AND el contenido de text difiere.

Después de detectar cambios, actualizar SOLO el campo last_reported_at en el snapshot. No sobreescribir el snapshot completo en un ciclo normal — eso solo ocurre con --reset.
Contrato del snapshot (no cambiar esta estructura):
json{
  "presentation_id": "string",
  "presentation_title": "string",
  "last_reported_at": "ISO8601 timestamp",
  "slides": [
    {
      "slide_object_id": "string",
      "slide_index": "integer (solo informativo, nunca usar para comparar)",
      "title": "string or null",
      "text_elements": [
        {
          "object_id": "string",
          "text": "string"
        }
      ]
    }
  ]
}
Tests requeridos para tests/test_slide_diff.py:
Escribir tests unitarios que cubran exactamente estos 5 casos, usando fixtures en memoria (sin llamadas reales a la API):

Sin cambios — snapshot y estado actual idénticos → retorna lista vacía de cambios.
Slide reordenada — mismos objectIds pero en distinto orden → retorna lista vacía (NO debe reportar cambio).
Slide eliminada — un slide_object_id del snapshot no aparece en el estado actual → reporta exactamente 1 cambio de tipo slide_removed.
Slide agregada — un slide_object_id nuevo aparece en el estado actual que no estaba en el snapshot → reporta exactamente 1 cambio de tipo slide_added.
Texto modificado — mismo slide_object_id y mismo text_element.object_id, pero texto diferente → reporta exactamente 1 cambio de tipo text_modified con campos before y after.


BUG 2 — Autenticación duplicada (resolver después del Bug 1)
Archivos: drive_monitor.py, slide_diff.py, main.py
Problema: drive_monitor.py y slide_diff.py cada uno autentica por separado con Google. Esto es frágil y genera dos service objects independientes donde debería haber uno.
Cómo debe funcionar:
Refactorizar para que main.py sea el único lugar donde se crea el objeto autenticado de Google. Debe crear DOS service objects al inicio:
pythondrive_service = build('drive', 'v3', credentials=creds)
slides_service = build('slides', 'v1', credentials=creds)
Luego pasar drive_service como parámetro a las funciones de drive_monitor.py, y slides_service como parámetro a las funciones de slide_diff.py. Ninguno de los dos módulos debe autenticar por su cuenta.
Tests requeridos:
Escribir tests que verifiquen que drive_monitor.py y slide_diff.py aceptan el service object como parámetro y no llaman a ninguna función de autenticación internamente. Usar unittest.mock.patch para confirmar que google.oauth2 o google_auth_oauthlib no son invocados dentro de esos módulos.
